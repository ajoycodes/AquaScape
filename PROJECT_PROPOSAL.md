# AquaScape — Project Proposal

## 1. Summary

**AquaScape** is a full-stack e-commerce and management platform for an aquarium retail business. It supports two user groups: **staff/admin** (who manage products, inventory, orders, and suppliers) and **customers** (who browse the shop, build custom aquarium setups, and place orders).

The system is built on an Oracle Database backend with a Node.js/Express API layer and a React frontend. All critical business logic — order placement, inventory updates, aquarium validation, and supplier workflows — is implemented as Oracle PL/SQL stored procedures and functions, ensuring data integrity at the database level.

## 4. Features

### 4.1 Product Catalogue
- Five product types: **Fish**, **Plant**, **Tank**, **Equipment**, **Decoration**
- Per-product fields: SKU, unit price, cost price, image URL, category, active flag
- Type-specific attributes: species data for fish, light requirements for plants, volume and dimensions for tanks, power and suitability for equipment

### 4.2 Inventory Management
- Real-time stock tracking with `qty_on_hand` and `qty_reserved` separation
- Configurable reorder level and reorder quantity per product
- Automatic **low-stock alerts** generated by trigger when stock falls at or below reorder level
- Alert acknowledgment workflow (resolve with user/timestamp)
- Immutable **inventory movement ledger** — every stock change (SALE, PURCHASE, RETURN, ADJUSTMENT, DAMAGE) is recorded with `quantity_delta` and `qty_after` snapshot

### 4.3 Aquarium Builder
- Customers design custom aquarium setups by selecting a tank and adding fish, plants, equipment, and decorations
- Three server-side **validation functions** run on save:
  - **Capacity validation** — fish stocking density vs tank volume
  - **Water type validation** — all items must match setup water type (FRESHWATER / SALTWATER / BRACKISH)
  - **Temperature validation** — fish temperature ranges must overlap
- **Compatibility rules** system — admin-defined INCOMPATIBLE / REQUIRES / NEUTRAL rules between products
- Setups can be saved as public share links or converted directly to orders

### 4.4 Shopping Cart & Orders
- One persistent cart per customer
- Order placement via `place_order` procedure — full ACID transaction covering:
  - Stock availability check
  - Discount code validation (date range, usage limit, minimum order amount)
  - Tax calculation (6%)
  - Cart-to-order conversion
  - Inventory deduction
  - Cart clearance
- Order status state machine: `PENDING → CONFIRMED → PROCESSING → SHIPPED → DELIVERED` (or `CANCELLED` / `REFUNDED`)
- Payment tracking per order (CASH, CARD, BANK_TRANSFER, E_WALLET)
- Return request and processing workflow with selective restocking of GOOD-condition items

### 4.5 Supplier & Purchase Orders
- Supplier directory with contact details and payment terms
- Purchase order workflow: `DRAFT → SUBMITTED → APPROVED → SHIPPED → RECEIVED`
- PO receiving records actual quantities delivered vs ordered
- Batch traceability on received stock (expiry date, cost, quantity)
- Receiving automatically calls `update_inventory` (move_type = PURCHASE)

### 4.6 Customer Management
- Customer profiles with contact details
- Lifetime value calculated via aggregation over order history
- Per-customer order history and return history

### 4.7 Reporting & Analytics
- **Dashboard KPIs**: total revenue, orders today, low-stock count, active customers
- **Monthly sales** chart (revenue, units, average order value)
- **Best sellers** by revenue and by units sold
- **Profit analysis** by product (revenue, COGS, gross margin %)
- **Fast movers**: 30-day sales velocity for demand planning
- **Audit log** viewer — every INSERT, UPDATE, DELETE on core tables
- **Stock movement log** — detailed ledger with reference tracking
- **Oracle objects browser** — live inventory of all DB objects (tables, views, procedures, triggers, functions)

### 4.8 Admin Tooling
- Compatibility rule management
- Inventory adjustment (manual stock correction with reason)
- Role-based access structure (ADMIN, STAFF, CUSTOMER)

## 7. Stored Procedures & Functions

### Procedures

| Procedure | Module | Description |
|---|---|---|
| `update_inventory` | Inventory | Central stock update — validates, adjusts `qty_on_hand`, records in movement ledger |
| `add_product_to_inventory` | Inventory | Creates inventory row for a new product |
| `create_aquarium_setup` | Builder | Creates a new DRAFT aquarium setup; returns `setup_id` OUT |
| `add_item_to_setup` | Builder | Adds/upserts product in setup; checks compatibility; validates product_type matches item_type |
| `save_setup` | Builder | Runs all 3 validations, marks setup SAVED, creates saved_setups record; returns `saved_id` OUT |
| `place_order` | Orders | Full ACID cart-to-order: stock check, discount, tax, order creation, inventory deduction, cart clear; returns `order_id` OUT |
